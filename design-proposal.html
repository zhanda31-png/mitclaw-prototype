八、技术实现要点
🔧 AI 推导引擎：基于岗位说明书文本，使用 LLM 提取结构化信息（场景、能力、受众），然后通过规则引擎匹配预置的知识库和 Skill 包。推导结果需要人工确认，避免 AI 幻觉导致错误配置。
模块 技术方案 关键点
文本解析 LLM + Prompt Engineering 从自然语言中提取：场景列表、能力要求、受众范围、复杂度评级
知识匹配 向量检索 + 标签匹配 根据岗位类型和场景，从知识库中检索最相关的预置知识包
Skill 推荐 规则引擎 + 场景映射 建立"场景→Skill"映射表，根据提取的场景自动推荐 Skill 包
策略生成 规则引擎 + 默认值 根据岗位复杂度（简单/中等/复杂）选择会话策略和记忆策略
试运行方案 组织架构 + 规则 默认选择申请人所在部门的 3-5 人，业务可调整
九、AI 推导引擎设计
从岗位说明书到完整配置方案的智能推导逻辑
🧠 核心思路：LLM 负责"理解语义"（提取场景、能力、受众），规则引擎负责"匹配配置"（知识库、Skill、策略）。两者结合，既保证理解深度，又保证配置准确性。
推导流水线（Pipeline）
岗位说明书 → LLM 语义解析 → 结构化输出 → 规则匹配 → 配置方案

1 LLM 语义解析层

将自然语言岗位说明书解析为结构化 JSON，提取 6 个维度的信息

输入：自然语言文本
"我需要一个客服虾，负责处理用户问题咨询和投诉，主要场景是回答产品使用问题、处理退换货申请..."
输出：结构化 JSON
{
  "name": "客服虾",
  "scenarios": ["产品咨询", "退换货", "投诉处理"],
  "capabilities": ["问答", "工单", "情绪识别"],
  "complexity": "medium",
  "serviceScope": "customer_service_team"
}
Prompt 设计要点
• 角色设定：你是岗位虾配置专家，负责从岗位说明书中提取关键信息
• 输出格式：严格输出 JSON，包含 name / scenarios / capabilities / complexity / serviceScope
• scenarios 提取：从"职责描述"、"工作场景"、"主要任务"等段落中提取动宾短语（如"处理投诉"、"审核文档"）
• capabilities 推导：根据场景推导所需能力（如"投诉"→情绪识别+工单处理，"审核"→规则校验+文本分析）
• complexity 评级：根据场景数量、能力复杂度、是否涉及多系统联动，输出 simple / medium / complex

2 知识库匹配引擎

根据岗位类型和场景，从预置知识库中检索最相关的知识包

匹配策略 逻辑 示例
岗位类型匹配 根据 LLM 提取的岗位类型（客服/HR/财务/研发），匹配预置的岗位知识包 客服虾 → 产品手册 + 退换话术 + FAQ
场景关键词匹配 对每个场景做关键词提取，与知识库标签做交集匹配 "退换货" → 匹配"退换货政策"知识库
向量相似度匹配 将场景描述做 embedding，与知识库摘要做余弦相似度排序，取 top-N 场景描述与"劳动法知识"相似度 0.82 → 推荐
部门知识继承 自动继承所在部门/空间的共享知识库 集团信息技术部 → 自动继承"IT规范"知识库
⚠️ 兜底策略：如果匹配结果为 0，AI 应主动询问"是否需要上传专属知识库？"，而不是静默跳过。知识库是虾的核心能力来源，不可省略。

3 Skill 推荐引擎

建立"场景 → Skill"映射表，根据提取的场景自动推荐 Skill 包

映射表示例
场景关键词推荐 Skill优先级
咨询/问答/回答智能问答 Skill必选
投诉/情绪/不满情绪识别 Skill推荐
工单/流程/审批工单系统对接 Skill必选
文档/报告/生成文档生成 Skill推荐
数据/分析/SQL数据分析 Skill推荐
代码/审查/PR代码审查 Skill必选
检索/搜索/查找知识检索 Skill必选
推荐策略
必选 Skill：场景强相关的 Skill，不勾选会影响核心功能
推荐 Skill：场景弱相关但能增强体验的 Skill
通识 Skill：所有虾默认挂载（如：智能问答、知识检索）
上限控制：单虾最多挂载 8 个 Skill，避免能力冲突

4 运行策略生成

根据岗位复杂度和场景特征，自动生成会话策略和记忆策略

会话策略决策树
if 场景包含"群聊协作":
→ 共享模式
elif 岗位类型 in [客服, HR, 财务]:
→ 隔离模式（独立处理）
elif complexity == "complex":
→ 混合模式
else:
→ 隔离模式（默认）
记忆策略决策树
岗位记忆（长期）：
→ 默认 开启（所有虾都需要积累经验）
会话记忆（短期）：
→ 默认 开启（保持对话连贯性）
事实轨迹（治理）：
if 岗位涉及"审批/决策/合规":
→ 开启（需审计追溯）
else:
→ 关闭（减少存储开销）

5 试运行方案生成

根据岗位类型和部门规模，自动生成试运行配置

参数 生成逻辑 默认值
试运行时长 根据 complexity 决定：simple=3天, medium=7天, complex=14天 7 天
试运行人群 默认：申请人 + 汇报对象 + 同部门 3-5 人 5 人
反馈收集 默认开启，试运行结束后自动收集群成员反馈 开启
达标标准 准确率 ≥ 80% + 用户满意度 ≥ 3.5/5 + 无 P0 故障 自动评估

6 置信度评分 & 兜底策略

每个推导结果附带置信度评分，低置信度项需人工确认

置信度评分规则
高 (≥80%)：直接展示，标记"AI 推荐"
中 (50-79%)：展示但标记"建议确认"
低 (<50%)：标记"需手动选择"，不自动填充
兜底策略
• 说明书过短（<50字）→ 提示"请补充岗位职责描述"
• 无法提取场景 → 使用通用 Skill 包 + 询问用户
• 知识库匹配为 0 → 询问"是否需要上传专属知识库？"
• 部门信息缺失 → 默认"全部成员"可见
